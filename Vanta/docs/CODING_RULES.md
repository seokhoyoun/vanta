# Vanta Coding Rules

작성일: 2026-03-14

## 목적

포팅과 리팩토링 과정에서 코드 스타일과 구현 방향을 일관되게 유지하기 위한 공통 규칙입니다.

## 기준 문서

기본 규칙으로 받아들이되, 현재 저장소 상태와 포팅 단계에 맞게 적용합니다.

## 기본 원칙

1. 가독성을 최우선으로 둡니다.
2. 짧은 코드보다 명확한 코드를 우선합니다.
3. 포팅 중에는 레거시 의미 보존과 디버깅 용이성을 우선합니다.
4. 기존 규칙과 충돌하지 않는 한 IDE 자동 포맷을 따릅니다.

## 문법 사용 규칙

1. `??` 문법 사용은 자제합니다.
2. 가능하면 람다 사용을 자제합니다.
3. 가능하면 LINQ 사용을 자제합니다.
4. `var` 사용은 기본적으로 자제합니다.
5. `new()` 축약 문법은 사용하지 않습니다.
6. `using` 선언문은 사용하지 않고 `using (...) { }` 문을 사용합니다.
7. 인라인 람다는 정말 짧은 한 줄 표현에만 제한적으로 허용합니다.
8. 컬렉션 변환, null 처리, 상태 전환은 한 줄 표현보다 단계적인 변수와 분기문을 우선합니다.

## 네이밍 규칙

1. 클래스, 구조체, 프로퍼티, public 메서드는 PascalCase를 사용합니다.
2. 지역 변수와 매개변수는 camelCase를 사용합니다.
3. 인터페이스는 `I` 접두사를 사용합니다.
4. enum은 `E` 접두사를 사용합니다.
5. 구조체는 `S` 접두사를 사용합니다. 단, `readonly record struct`는 예외로 둘 수 있습니다.
6. private 필드는 `m` 접두사를 사용합니다.
7. bool 변수는 `b` 접두사를 사용합니다.
8. bool 프로퍼티는 `Is`, `Has`, `Can`, `Should` 중 하나로 시작합니다.
9. 상수와 `static readonly` 상수 객체는 모두 대문자와 밑줄을 사용합니다.

## 함수 및 API 규칙

1. 메서드 이름은 가능한 한 동사 기준으로 짓습니다.
2. 값을 반환하는 함수는 무엇을 반환하는지 이름에서 드러나야 합니다.
3. getter, setter 메서드 대신 프로퍼티를 우선합니다.
4. 기본 매개변수보다 오버로드를 우선합니다.
5. `async void`는 이벤트 핸들러 외에는 사용하지 않습니다.
6. `async` 메서드 이름에 `Async` 접미사는 붙이지 않습니다.
7. public 함수는 가능한 한 `null`을 입력이나 반환으로 허용하지 않는 방향을 우선합니다.
8. `null` 반환이 필요한 경우 함수 이름 뒤에 `OrNull`를 붙이는 방식을 고려합니다.

## 제어 흐름 및 방어 코드 규칙

1. `switch`에는 항상 `default`를 둡니다.
2. 절대 도달하지 않아야 하는 `default`는 assert 또는 예외로 처리합니다.
3. 코드가 기대하는 모든 핵심 가정은 assert로 표현합니다.
4. 예외 처리는 가능한 한 경계 계층에서 수행합니다.
5. 내부 도메인 함수는 가능한 한 유효한 입력만 받는다고 가정하는 방향으로 설계합니다.
6. 변수 가리기(shadowing)는 허용하지 않습니다.

## 포맷팅 규칙

1. 중괄호는 항상 새 줄에서 엽니다.
2. 코드가 한 줄이어도 중괄호를 생략하지 않습니다.
3. 한 줄에 변수 하나만 선언합니다.
4. 클래스는 원칙적으로 파일 하나당 하나씩 둡니다.
5. 파일 이름은 타입 이름과 일치시킵니다.

## 컬렉션 및 자료형 규칙

1. `System.Collections`보다 `System.Collections.Generic`를 우선합니다.
2. 순수 배열 사용은 허용합니다.
3. 범용적인 숫자, 문자열 조합이 도메인 의미를 가지면 강타입화 후보로 봅니다.

## 직렬화 및 DTO 규칙

1. 직렬화 대상 DTO는 기본적으로 class를 우선합니다.
2. 직렬화 전용 attribute로 도메인 모델을 오염시키지 않는 방향을 우선합니다.
3. 직렬화 호출은 직접 흩뿌리지 말고 wrapper를 두는 방향을 우선합니다.

## 허용 예외

- 단순한 guard 성격의 null 처리
- 테스트 코드에서 가독성이 더 명확한 경우
- 반복문보다 더 명확하다고 합의된 매우 단순한 projection

기본 원칙은 "예외적으로 허용"이지 "자유롭게 사용"이 아닙니다.

## 현재 저장소와의 차이

아래 항목은 외부 기준과 현재 저장소 설정이 아직 완전히 일치하지 않습니다.

1. 현재 `Directory.Build.props`는 `ImplicitUsings=enable` 상태입니다.
2. 현재 일부 코드에는 `file-scoped namespace`를 사용 중입니다.

이 항목들은 추후 규칙 적용 단계에서 정리 여부를 별도로 결정합니다.

## 적용 범위

- `Vanta.Core`
- `Vanta.Comm.Contracts`
- `Vanta.Comm.Abstractions`
- `Vanta.Comm.Application`
- `Vanta.Comm.Infrastructure.*`
- `Vanta.Comm.Device.*`
- `Vanta.Comm.Process.*`

## 해석 원칙

- 절대 금지보다 "기본 금지, 필요 시 예외 허용" 기준으로 해석합니다.
- 판단이 애매하면 축약 문법보다 명시적 문법을 선택합니다.
